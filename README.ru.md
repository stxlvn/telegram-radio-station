# Telegram Radio Station

*[English version](README.md)*

Круглосуточное интернет-радио на основе музыкального архива Telegram-канала,
с синхронной ретрансляцией в видеочат Telegram и веб-плеером в стиле
Material Design 3.

## Возможности

**Вещание**

- Непрерывный поток в MP3 (128kbps) и lossless FLAC из одного источника,
  качество выбирается слушателем
- HLS-зеркало рядом с обычным потоком — даёт плееру реальную перемотку
  (назад на 10 сек, возврат к живому эфиру), чего не позволяет обычный
  потоковый поток без Content-Length
- Опциональный ICY-metadata прокси для FLAC/Ogg-потока — для аппаратных
  стримеров, которые читают только классические ICY-теги и никогда не
  перечитывают собственные цепочечные заголовки живого Ogg-потока;
  включается только для тех клиентов, кто его явно запросил, остальных не
  трогает
- Параллельная ретрансляция того же аудио в видеозвонок Telegram-группы по
  RTMP, с живым видеокадром (текущий трек, очередь, кнопка подписки),
  перерисовываемым на каждом тике, а не статичной заглушкой

**Автосоставление очереди**

- Треки выбираются случайно из всего аудиоархива Telegram-канала, без
  ручного ведения плейлиста
- Фильтр по длительности (отсеивает и короткие джинглы, и часовые сборники)
- Очистка артефактов виниловых рипов и треклистов в названии (маркеры
  сторон вида `A2`/`B1`, ведущие номера треков)
- Многоуровневое определение эмбиента/саундтрек-кью, чтобы фоновая
  инструментальная музыка не забивала ротацию, но при этом не задевала
  обычные песни, просто оказавшиеся на одном релизе с ней: хэштеговые
  конвенции канала, ключевые слова в названии, проверка жанра/стиля на
  Discogs (с подтверждением по треклисту релиза, а не только по его
  названию), которая отменяется, если рядом стоит обычный «песенный» жанр
  (рок/поп/хип-хоп и т.д.) или явно указан приглашённый вокалист, плюс
  фильтр интро/аутро/интервью/концертной записи по названию
- Дневная квота проигрываний на альбом, чтобы один релиз не занимал
  эфир часами
- Защита от повтора: один трек не может повториться раньше чем через 48 часов
- Каждое исключение логируется с конкретной причиной, а не просто ID
- Треки, загруженные через админ-панель, не попадают под автоматическую
  чистку устаревшей очереди

**Обогащение метаданных**

- Многоуровневый fallback для артиста/названия: теги в файле → собственные
  поля Telegram (title/performer) → посты-описания канала (хэштеги жанров,
  строки вида «Артист — Альбом») → поиск в Discogs как последний вариант
- Сопоставление трека в Discogs по длительности (а для поиска описания —
  по реальному наличию в треклисте) с полным треклистом релиза, а не по
  названию самого релиза — иначе треку могло присвоиться название альбома
  вместо реальной песни, либо корректное совпадение отбрасывалось бы
  только потому, что искомая песня не является заглавным треком
- Текст описания сначала пробуется через собственные заметки релиза на
  Discogs, затем через Wikipedia (сначала русская, при неудаче —
  английская), с проверками на страницы-дизамбиги, ложные совпадения с
  статьями о населённых пунктах и слабо связанные результаты поиска
- Обложка ищется тем же путём, с автоматическим откатом на аватар канала
- Данные для ещё не играющего трека подгружаются заранее — как только он
  доходит до позиции «скоро в эфире» в очереди, так что к моменту
  реального начала воспроизведения описание/обложка/ссылка на клип уже
  готовы, без задержки на живой запрос при каждой смене трека

**Клип на YouTube**

- Кнопка «Смотреть клип» появляется только когда для трека реально нашёлся
  подходящий видеоролик — ссылка, не встроенный плеер
- Поиск выполняется один раз на сервере при смене трека (не в браузере
  каждого слушателя) — иначе бесплатная квота YouTube Data API кончалась бы
  за несколько часов при частой смене треков
- Отсеивает аудио-заглушки: авто-сгенерированные каналы вида
  «Артист - Topic», ролики с пометками audio/lyric video и т.п.
- При исчерпании квоты официального API — автоматический fallback на
  yt-dlp (тоже только для поиска, само видео никуда не скачивается и не
  проксируется)

**Админ-панель**

- Веб-интерфейс с паролем для загрузки треков прямо в очередь, по одному
  или пачкой
- Вставка трека сразу после конкретного элемента очереди, а не только в конец
- Изменение порядка уже стоящих в очереди треков
- Пропуск текущего трека без нарушения остальной предзагруженной очереди
- Живое обновление статуса (текущий трек, что дальше) без перезагрузки
  страницы
- Индикатор прогресса загрузки для больших файлов на мобильном интернете

**Веб-плеер**

- Адаптивная вёрстка: на десктопе — три карточки в ряд (описание канала,
  плеер, очередь), на мобильном разворачивается в полноэкранный плеер по
  тапу, с отдельной раскладкой бок о бок для короткого/горизонтального
  экрана телефона
- На мобильном — нижняя панель вкладок в стиле Material Design 3
  («Радио» с плеером и очередью, «О канале» с описанием), а не длинный
  скролл через всё подряд; на десктопе не нужна — там всё и так лежит рядом
- Прогресс-бар синхронизирован с серверным временем, не сбивается от
  буферизации, с автопереподключением при обрыве связи — включая поправку
  показа смены трека на реальную дистанцию HLS-плеера от живого края
  эфира, чтобы отображаемый трек не убегал вперёд того, что реально слышно
- Интеграция с системным медиа-центром ОС — управление с экрана
  блокировки/шторки уведомлений, обложка, живой прогресс-бар на любой
  платформе с поддержкой Web Media Session API
- Переключатель качества (MP3/FLAC) с предупреждением о расходе трафика
- Счётчик подписчиков канала в реальном времени
- Полноэкранный просмотр обложки (пропорции сохраняются, не обрезается) с
  возможностью перевода описания на русский, если оно на другом языке
- Устанавливается как PWA (manifest + service worker) — установленное
  приложение получает от Android и iOS заметно больше прав на фоновое
  воспроизведение, чем обычная вкладка браузера, которую система со временем
  глушит при заблокированном экране; ненавязчивый баннер с объяснением и
  кнопкой установки (или инструкцией для iOS, где нет программного вызова
  установки) показывается один раз после начала воспроизведения

**Инфраструктура**

- Цепочка relay-серверов в нескольких регионах, чтобы поток оставался
  доступен при выборочных ограничениях маршрутизации, с автопереключением
  на резервный источник
- Сторожевой сервис, следящий за всеми компонентами (доступность потока,
  свежесть воспроизведения, состояние каждого systemd-юнита, здоровье
  аудио-входа самой RTMP-трансляции) и перезапускающий зависшее

## Структура проекта

```
backend/     Наполнение очереди, публикация текущего трека (+ предзагрузка
             метаданных), сборка списка очереди, ICY-metadata прокси,
             публикация числа слушателей, админ-панель, сторожевой сервис,
             разовые скрипты индексации/сканирования Telegram
liquidsoap/  Граф источников радио, кроссфейд, вывод в Icecast/HLS
tgstream/    Ретрансляция в Telegram по RTMP (рендер видеокадра,
             кодирование ffmpeg, обновление счётчика подписчиков)
frontend/    Веб-плеер (одна страница, без сборки)
nginx/       Конфиги реверс-прокси (источник + edge-relay)
systemd/     Юниты для всех фоновых сервисов
deploy.sh    Быстрая установка на чистый VPS: системные пакеты, актуальный
             Liquidsoap, venv, systemd-юниты — сознательно не запускает
             сервисы и не заполняет секреты, в конце печатает чек-лист
             оставшихся шагов
```

## Установка

Быстрый путь на чистом Ubuntu/Debian VPS: `sudo ./deploy.sh`, затем следуйте
чек-листу, который он выведет в конце (креды Telegram, пароль Icecast,
домен/nginx/SSL, разовая сборка индекса треков, запуск сервисов по порядку).

Вручную:

1. Скопируйте `.env.example` в `/etc/musicbestman/env` и заполните реальными
   значениями (креды Telegram API, пароль Icecast, RTMP URL, опционально
   токены Discogs и YouTube Data API — см. комментарии в файле, что
   обязательно, а что нет).
2. Установите systemd-юниты из `systemd/` и скрипт Liquidsoap из
   `liquidsoap/`. Нужен Liquidsoap >= 2.4 — версия из apt на Ubuntu обычно
   заметно отстаёт; при необходимости возьмите актуальный релиз со
   [страницы релизов Savonet](https://github.com/savonet/liquidsoap/releases).
3. Замените плейсхолдеры IP-адресов/домена в `nginx/` на свои сервера.
4. Авторизуйте Telegram-сессию для аккаунта, от имени которого работают
   скрипты (Telethon запросит это при первом запуске) — итоговый файл
   `.session` не должен попадать в систему контроля версий, как и любой
   другой секрет.
5. Соберите начальный индекс треков (`backend/tg_index_build.py`) прежде
   чем впервые запускать `queue-filler.service`.
